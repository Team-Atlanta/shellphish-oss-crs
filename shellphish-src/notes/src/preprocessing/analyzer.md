# Analyzer

## Summary

The Analyzer phase extracts metadata from build artifacts and identifies harnesses for vulnerability analysis. It consists of four main components: **Target Identifier** (harness detection), **Configuration Splitter** (build matrix generation), **CodeQL Analysis** (program graph extraction), and **SARIF Processing** (report validation). This phase bridges the gap between raw build artifacts and structured metadata required for vulnerability discovery.

**Scope Definition:** The whitepaper describes "Analyzer" as a conceptual phase for metadata extraction. In this documentation, we group the following components under "Analyzer" based on their shared purpose: they all consume Bob the Builder's outputs and produce structured metadata for downstream vulnerability analysis, without performing actual vulnerability discovery themselves.

**Execution Model:** These components do NOT run strictly sequentially:

- **Target Identifier**: Runs first, depends only on base project metadata and build artifacts
- **Configuration Splitter**: Starts after Target Identifier for harness info generation, but build configuration splitting can start earlier
- **CodeQL Analysis**: Runs in PARALLEL with the above components (only needs base metadata, not augmented metadata) - this is a deliberate optimization for time-limited competition runs
- **SARIF Processing**: Triggered independently when SARIF reports become available

> From whitepaper [Section 4.2](https://github.com/sslab-gatech/shellphish-afc-crs/blob/main/notes/src/whitepaper/Artiphishell-3.md#42-analyzer):
>
> After Bob the Builder creates the artifacts previously described, the Analyzer agent extracts metadata about the target and stores it in the Analysis Graph database. The Analyzer agent is also responsible for the identification of the endpoints that represent harnesses and will be used in the vulnerability analysis process.
>
> From whitepaper [Section 3.3](https://github.com/sslab-gatech/shellphish-afc-crs/blob/main/notes/src/whitepaper/Artiphishell-3.md#33-preprocessing):
>
> Note that while the results of this initial preprocessing step are used by several subsequent analyses, there are agents that will start their job with limited information (i.e., before some of the preprocessing is terminated), in an attempt to optimize time and resources.

## Data Flow

```text
Build Artifacts (from Bob the Builder)
    ↓
Target Identifier
    ├─ Scan source tree
    ├─ Detect harnesses (C/C++: llvm-symbolizer, Java: regex + search)
    ├─ Categorize files by type
    └─ Output: AugmentedProjectMetadata
    ↓
Configuration Splitter
    ├─ Generate arch × sanitizer matrix
    ├─ Create BuildConfiguration objects
    ├─ (Optional) Split harnesses per configuration
    └─ Output: Build configs + Harness infos
    ↓
CodeQL Analysis (Parallel)
    ├─ Build: Create CodeQL database (HEAD and HEAD~1)
    ├─ Analysis Graph: Extract CFG/DFG/CPG
    ├─ Quickseed: Identify vulnerability sinks
    └─ CWE Queries: Run security-specific queries
    ↓
SARIF Processing
    └─ Validate and store SARIF reports
    ↓
Output Repositories
    ├─ augmented_project_metadatas
    ├─ project_build_configurations
    ├─ project_harness_infos
    ├─ codeql_reports
    └─ quickseed_codeql_reports
```

**Orchestration**: The PyDataTask framework automatically launches tasks when their input dependencies are available. Each component is defined in a `pipeline.yaml` with task dependencies, input/output repositories, and execution containers.

## Components

### 1. Target Identifier

**Purpose:** Detect harness entry points and categorize source files by type.

**Implementation:** [components/target-identifier/analyze_target.py](https://github.com/sslab-gatech/shellphish-afc-crs/blob/main/components/target-identifier/analyze_target.py)

#### Harness Detection

**For C/C++ Projects** ([Lines 158-182](https://github.com/sslab-gatech/shellphish-afc-crs/blob/main/components/target-identifier/analyze_target.py#L158-L182)):

- Parses `<harness>.shellphish_harness_symbols.json` files (generated by llvm-symbolizer during build)
- Locates `LLVMFuzzerTestOneInput` function in symbol table
- Extracts `SourceLocation` with file path, line, column, and function name
- Returns exact source location for fuzzing entry point

**For Java Projects** ([Lines 183-296](https://github.com/sslab-gatech/shellphish-afc-crs/blob/main/components/target-identifier/analyze_target.py#L183-L296)):

- Extracts `--target_class` parameter from harness wrapper via regex
- Converts class path to file path (e.g., `com.foo.Bar` → `com/foo/Bar.java`)
- Searches for `fuzzerTestOneInput` method in source tree
- Falls back to edit-distance matching ([Lines 135-155](https://github.com/sslab-gatech/shellphish-afc-crs/blob/main/components/target-identifier/analyze_target.py#L135-L155)) using Levenshtein algorithm when exact match fails
- Populates `JavaInfo` metadata (class path, package, method descriptor)

#### File Categorization

The [`identify_targets`](https://github.com/sslab-gatech/shellphish-afc-crs/blob/main/components/target-identifier/analyze_target.py#L36-L133) function walks the source tree and categorizes files by extension into 30+ types:

**Programming Languages:** c, cpp, java, python, go, ruby, javascript

**Build Systems:** bazel, autotools, batch

**Data Formats:** json, yaml, xml, markdown

**Archives:** zip, tar, jar

**Other:** certificates, images, office documents

Results stored in `files_by_type` dictionary with file counts per category.

#### Output

Generates `AugmentedProjectMetadata` ([Line 339](https://github.com/sslab-gatech/shellphish-afc-crs/blob/main/components/target-identifier/analyze_target.py#L339)) containing:

- `harness_source_locations` - Map of harness name → SourceLocation
- `known_sources` - Detected known libraries/frameworks
- `files_by_type` - File statistics by category
- Original project metadata merged with shellphish-specific fields

### 2. Configuration Splitter

**Purpose:** Generate build configuration matrix for different architectures and sanitizers.

**Implementation:** [components/configuration-splitter/split_configurations.py](https://github.com/sslab-gatech/shellphish-afc-crs/blob/main/components/configuration-splitter/split_configurations.py)

#### Build Configuration Generation

**Supported Matrix** ([Lines 16-17](https://github.com/sslab-gatech/shellphish-afc-crs/blob/main/components/configuration-splitter/split_configurations.py#L16-L17)):

- **Architectures:** x86_64 (only)
- **Sanitizers:** address, undefined, memory, thread

**Process** ([Lines 54-72](https://github.com/sslab-gatech/shellphish-afc-crs/blob/main/components/configuration-splitter/split_configurations.py#L54-L72)):

1. Iterates over project metadata architectures and sanitizers
2. Filters to supported combinations only
3. Creates `BuildConfiguration` model for each combination
4. Uploads to `project_build_configurations` repository with deduplication
5. Returns configuration key for downstream reference

#### Harness Splitting

**When enabled** (`--include-harness-splitting` flag, [Lines 81-123](https://github.com/sslab-gatech/shellphish-afc-crs/blob/main/components/configuration-splitter/split_configurations.py#L81-L123)):

1. **Project Harness Metadata** ([Lines 89-101](https://github.com/sslab-gatech/shellphish-afc-crs/blob/main/components/configuration-splitter/split_configurations.py#L89-L101)):
   - Creates `ProjectHarnessMetadata` for each harness
   - Includes harness name and binary path (`out/<harness>`)
   - Uploaded to `project_harness_metadatas` repository

2. **Harness Info Matrix** ([Lines 104-118](https://github.com/sslab-gatech/shellphish-afc-crs/blob/main/components/configuration-splitter/split_configurations.py#L104-L118)):
   - Cross-product of configurations × harnesses
   - Creates `HarnessInfo` combining build config + harness metadata
   - Links `build_configuration_id` with `project_harness_metadata_id`
   - Uploaded to `project_harness_infos` repository

#### Output Metadata

Generates `target_split_metadata` ([Lines 74-126](https://github.com/sslab-gatech/shellphish-afc-crs/blob/main/components/configuration-splitter/split_configurations.py#L74-L126)) containing:

- `num_build_configs` - Total configuration count
- `build_config_keys` - List of configuration IDs
- `build_configurations` - Full configuration objects
- `num_harnesses` / `num_harness_infos` - Harness statistics
- `harness_info_keys` - Combined harness-config IDs

### 3. CodeQL Analysis

**Purpose:** Extract program structure (CFG/DFG/CPG) and identify potential vulnerabilities.

**Implementation:** [components/codeql/pipeline.yaml](https://github.com/sslab-gatech/shellphish-afc-crs/blob/main/components/codeql/pipeline.yaml)

#### Database Creation

**codeql_build Task** ([Lines 191-334](https://github.com/sslab-gatech/shellphish-afc-crs/blob/main/components/codeql/pipeline.yaml#L191-L334)):

- Builds project with CodeQL instrumentation (`shellphish_codeql`)
- Creates database at `artifacts/work/sss-codeql-database.zip`
- Supports buildless database creation (no build tracing)
- Uploads database to shared storage via `codeql-upload-db` command
- Priority: 20B (same as canonical build)
- Timeout: 180 minutes

**Delta Mode Support - codeql_build_base** ([Lines 64-189](https://github.com/sslab-gatech/shellphish-afc-crs/blob/main/components/codeql/pipeline.yaml#L64-L189)):

- Builds HEAD~1 version for differential analysis
- Uses `--git-ref "HEAD~1"` parameter
- Creates separate base database for comparison
- Only runs when `delta_mode_task` input exists

#### Analysis Queries

**codeql_analysis_graph Task** ([Lines 336-424](https://github.com/sslab-gatech/shellphish-afc-crs/blob/main/components/codeql/pipeline.yaml#L336-L424)):

- Executes custom analysis queries via `analysis_query.py`
- Extracts call graph, CFG, DFG, CPG
- Requires function indices for context (`full_functions_indices`)
- Timeout: 60 minutes
- Resources: 4 CPU, 8Gi RAM

**quickseed_codeql_query Task** ([Lines 426-523](https://github.com/sslab-gatech/shellphish-afc-crs/blob/main/components/codeql/pipeline.yaml#L426-L523)):

- Language-specific vulnerability queries:
  - **Java:** Identifies injection sinks (SQL, XPath, Command, XXE, etc.)
  - **C/C++:** Dumps reachability and call graph information
- Outputs sink locations to `quickseed_codeql_report`
- Creates `discovery_vuln_reports` for downstream analysis
- Priority: 100 (lower than build)

**codeql_cwe_queries Task** ([Lines 525-621](https://github.com/sslab-gatech/shellphish-afc-crs/blob/main/components/codeql/pipeline.yaml#L525-L621)):

- Runs CWE-specific security queries
- Generates SARIF reports (`codeql_cwe_sarif_report`)
- Produces structured vulnerability metadata (`codeql_cwe_report`)
- Includes vulnerable function mapping and rule statistics
- Resources: 8 CPU, 12Gi RAM

#### Output Formats

**SARIF Reports:** Standard format with version 2.1.0, includes rules and results

**Quickseed Report:** JSON with sinks categorized by vulnerability type

**CWE Report:** Metadata including total vulnerabilities and findings per rule

### 4. SARIF Processing

**Purpose:** Validate and process static analysis reports.

**Implementation:** [components/sarifguy/pipeline.yaml](https://github.com/sslab-gatech/shellphish-afc-crs/blob/main/components/sarifguy/pipeline.yaml)

**Process:**

- Receives SARIF reports from CodeQL or external sources
- Validates report structure and content
- Stores processed reports in `sarif_reports` repository
- Metadata tracked in `sarif_metadatas` for downstream consumption

## Error Handling

### Harness Detection Failures

- C/C++: Asserts exactly one symbol entry for harness
- Java: Multiple fallback strategies with edit-distance matching
- Returns partial SourceLocation if no exact match found
- Controlled by `artiphishell_should_fail_on_error()` flag

### Build Failures

- CodeQL build uses exit trap: `exit 33` triggers failure, others succeed
- Empty reports created at task start to ensure valid output
- Trap handlers return 0 on normal exit

### Telemetry

- OpenTelemetry spans track execution: `analyze_target`, `function_index_generator`
- Events logged for key operations (result writing, analysis completion)
- Status tracking via `status_ok()` helper

## Note on Whitepaper vs. Implementation

The [whitepaper Section 4.2](https://github.com/sslab-gatech/shellphish-afc-crs/blob/main/notes/src/whitepaper/Artiphishell-3.md#42-analyzer) lists additional items under "Analyzer" that are actually implemented as separate components in the actual architecture:

### WLLVMbear

**Whitepaper description**: "Contains all the compiler commands executed during the building of the various artifacts."

**Actual implementation**: This is part of the **Bob the Builder** phase, not Analyzer. WLLVMbear/wllvm is a build-time tool that wraps compiler commands during compilation to extract compilation database information and enable instrumentation. The compilation metadata is generated during the build process and consumed by downstream components.

### Seed Corpus

**Whitepaper description**: "An initial pre-determined corpus of input seeds based on popular formats and protocols."

**Actual implementation**: This is handled by a separate component called **Corpus-Guy** ([components/corpus-guy/pipeline.yaml](https://github.com/sslab-gatech/shellphish-afc-crs/blob/main/components/corpus-guy/pipeline.yaml)), which runs as an independent pipeline task. Corpus-Guy:

- Depends on Analyzer outputs (augmented project metadata, harness info)
- Has three main tasks:
  - `corpus_diff_splitter` - Splits diff chunks as seeds for delta mode
  - `corpus_inference_llm` - Uses LLM to infer input formats and generate corpus
  - `corpus_kickstart` - Kickstart initial corpus generation from known formats
- Produces `likely_input_formats_corpuses`, dictionaries, and grammars
- Syncs generated seeds to fuzzer instances via shared storage (`/shared/fuzzer_sync`)

The whitepaper groups related preprocessing activities conceptually, while the actual implementation has a more modular architecture where corpus generation is a separate downstream component that consumes Analyzer outputs.
